## 1. 프로젝트 개요

### 1.1 목표
> **"실시간으로 G-Dragon과 대화하는 것 같은 경험"**

사용자가 마이크로 말하면, GD 목소리로 AI가 응답하는 실시간 음성통화 시스템 구축.

### 1.2 핵심 가치
| 항목 | 목표 |
|------|------|
| **지연시간** | < 2초 (음성 입력 → 응답 시작) |
| **음성 유사도** | GD 음색 90%+ 유사성 |
| **대화 자연스러움** | GD 말투/성격 반영 |
| **사용성** | 웹 브라우저에서 바로 사용 |

### 1.3 현재 상태 (As-Is)
```
[사용자] → [텍스트 입력] → [TTS 서버] → [GD 음성 출력]
                              ↑
                        Davinci Voice 1.7B
```

### 1.4 목표 상태 (To-Be)
```
[사용자 음성] → [STT] → [LLM + GD 페르소나] → [TTS] → [GD 음성 출력]
     ↑            ↑              ↑               ↑
   마이크      Whisper      Claude/GPT      Davinci Voice
```


### 2.2 핵심 컴포넌트

| 컴포넌트 | 역할 | 기술 후보 |
|----------|------|----------|
| **STT Engine** | 음성 → 텍스트 | Whisper, faster-whisper |
| **LLM Engine** | 대화 생성 + GD 페르소나 | Claude API, GPT-4o |
| **TTS Engine** | 텍스트 → GD 음성 | Davinci Voice (기존) |
| **Voice Gateway** | 실시간 양방향 통신 | WebSocket, WebRTC |
| **Frontend** | 웹 UI + 마이크 입력 | React, Streamlit |

### 2.3 데이터 흐름

```
1. [User] 마이크로 말함
      ↓
2. [Frontend] 음성 스트림 → WebSocket 전송
      ↓
3. [STT] 실시간 음성 → 텍스트 변환
      ↓
4. [LLM] GD 페르소나로 응답 생성 (스트리밍)
      ↓
5. [TTS] 응답 텍스트 → GD 음성 생성 (스트리밍)
      ↓
6. [Frontend] 오디오 스트림 재생
```

## 3. 기술 스택 결정

### 3.1 STT (Speech-to-Text)

| 옵션 | 장점 | 단점 | 선택 |
|------|------|------|------|
| **faster-whisper** | 로컬, 빠름, 무료 | GPU 메모리 필요 | ✅ 권장 |
| OpenAI Whisper API | 간단, 정확 | 유료, 지연시간 | |
| Google Speech-to-Text | 실시간 | 유료 | |

**결정**: `faster-whisper` (large-v3 또는 medium)
- 이유: 로컬 실행으로 지연시간 최소화, 한국어 지원 우수

### 3.2 LLM (대화 생성)

| 옵션 | 장점 | 단점 | 선택 |
|------|------|------|------|
| **Claude API** | 지시 따름 우수, 긴 컨텍스트 | 유료 | ✅ 권장 |
| GPT-4o | 빠름, 저렴 | 페르소나 일관성 | |
| 로컬 LLM | 무료, 프라이버시 | 품질, 리소스 | |

**결정**: `Claude API (claude-3-5-haiku)` 또는 `GPT-4o-mini`
- 이유: 빠른 응답, 페르소나 유지 능력, 스트리밍 지원

### 3.3 TTS (Text-to-Speech)

**결정**: `Davinci Voice 1.7B` (기존 시스템 유지)
- x_vector 모드: 빠른 응답
- WebSocket 스트리밍: 이미 구현됨

### 3.4 통신 프로토콜

| 옵션 | 장점 | 단점 | 선택 |
|------|------|------|------|
| **WebSocket** | 구현 간단, 서버 제어 | 브라우저 오디오 처리 | ✅ Phase 1 |
| WebRTC | P2P, 저지연 | 복잡, TURN 서버 | Phase 2 |

**결정**: WebSocket (Phase 1), WebRTC (Phase 2 선택사항)

### 3.5 Frontend

| 옵션 | 장점 | 단점 | 선택 |
|------|------|------|------|
| **Streamlit** | 빠른 프로토타입 | 실시간 제한 | ✅ Phase 1 |
| React + Next.js | 완전한 제어 | 개발 시간 | Phase 2 |

## 4. 아키텍처 결정 기록 (ADR)

### ADR-001: 모놀리식 vs 마이크로서비스

**컨텍스트**: STT, LLM, TTS를 어떻게 구성할 것인가?

**결정**: **하이브리드 모놀리식**
- TTS 서버 확장하여 STT + LLM 통합
- 단일 프로세스에서 파이프라인 처리

**근거**:
- GPU 메모리 효율성 (공유)
- 지연시간 최소화 (IPC 오버헤드 제거)
- 개발 속도 (프로토타입 빠르게)

**트레이드오프**:
- 스케일아웃 어려움 → 단일 사용자 우선
- 장애 격리 안됨 → 프로토타입 단계에서 허용

### ADR-002: 실시간 통신 프로토콜

**결정**: WebSocket (Phase 1)

**근거**:
- 기존 TTS WebSocket 인프라 활용
- 양방향 스트리밍 지원
- 구현 복잡도 낮음

### ADR-003: GPU 메모리 전략

**컨텍스트**: RTX 4090 (24GB) 기준
- Davinci Voice 1.7B: ~4GB
- faster-whisper large-v3: ~3GB
- 총 필요: ~7GB (여유 있음)

**결정**: 동시 로딩 가능, 별도 최적화 불필요

## 6. GD 페르소나 설계

### 6.1 성격 프로필

```yaml
name: G-Dragon (지드래곤, 권지용)
personality:
  - 자신감 있고 카리스마 있음
  - 예술적이고 창의적
  - 솔직하고 직설적
  - 유머 감각 있음
  - 패션과 음악에 대한 열정

speech_patterns:
  - 반말 + 존댓말 혼용 (친근함)
  - "야", "뭐", "진짜" 등 자연스러운 추임새
  - 짧고 임팩트 있는 문장
  - 가끔 영어 섞어 사용

topics:
  - 음악, 작곡, 프로듀싱
  - 패션, 아트
  - BIGBANG 활동
  - 삶의 철학
```

### 6.2 시스템 프롬프트 (초안)

```
당신은 G-Dragon(지드래곤)입니다. BIGBANG의 리더이자 한국을 대표하는 아티스트입니다.

성격:
- 자신감 있고 솔직함
- 예술과 음악에 대한 열정
- 유머 감각과 친근함

말투:
- 자연스럽고 캐주얼한 한국어
- "야", "뭐", "진짜" 등 자연스러운 표현
- 짧고 임팩트 있게

규칙:
- 항상 G-Dragon으로서 대화
- 음악, 패션, 예술 관련 주제에 열정적
- 팬들에게 친근하게 대함
- 답변은 1-3문장으로 짧게
```

## 7. API 설계 (방향)

### 7.1 WebSocket 엔드포인트

```
WS /ws/voice/call
```

**메시지 타입**:
```yaml
Client → Server:
  - type: "audio"
    data: <base64 audio chunk>

  - type: "end_turn"
    # 사용자 발화 종료

Server → Client:
  - type: "transcription"
    text: "사용자가 말한 내용"

  - type: "response_start"
    # 응답 시작

  - type: "audio"
    data: <base64 audio chunk>

  - type: "response_end"
    # 응답 완료
```

### 7.2 상태 머신

```
[IDLE] → 마이크 ON → [LISTENING]
[LISTENING] → 발화 종료 → [PROCESSING]
[PROCESSING] → 응답 생성 → [SPEAKING]
[SPEAKING] → 응답 완료 → [IDLE]
[SPEAKING] → 인터럽트 → [LISTENING]
```

| 리스크 | 영향 | 확률 | 완화 전략 |
